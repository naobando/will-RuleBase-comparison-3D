## Q1: 要因の修正は想定より時間がかかったものか？

はい。カメラ変更により以前は問題にならなかった部分が顕在化し、複数の要因が重なっていたため、1つずつ切り分けて対処する必要がありました。

要因は大きく分けて2種類あります。

### カメラ変更で顕在化した要因（画像の見え方が変わったことで露呈）

| 要因 | 内容 | なぜカメラ変更で露呈したか |
| --- | --- | --- |
| 透視変換・シアー歪み | findHomography(8自由度)がアスペクト比を歪ませていた | 以前のカメラでは画角が近く歪みが目立たなかった。新カメラで画角差が大きくなり顕在化 |
| 固定座標cropの位置ズレ | 固定値(x:270,y:550)が新カメラの撮影位置と合わない | カメラ位置・画角が変わり部品の写る位置が変わった |
| 前景マスクの穴 | 部品内部の暗い領域が背景と誤判定 | 新カメラの輝度特性で内部の暗さが変わった |
| ECC移動量上限 | 固定20pxの上限が新カメラの画像サイズに対して厳しすぎた | 画像サイズ・解像度が変わった |

### 元々あったが気づいていなかった要因

| 要因 | 内容 |
| --- | --- |
| 前景マスクの背景干渉 | マスターのみで前景マスクを作っていたため境界で偽差分が発生 |
| BBOXの統合しすぎ・分割しすぎ | E+SIFTパスにマージ/帯フィルタがなかった |
| アプリとスクリプトの結果不一致 | crop_fixed_paramsがアプリに渡されていなかった |

これらは以前から存在していたが、カメラ変更で精度が全体的に下がったことで調査する過程で発見されました。

## Q2: 水曜夕会後のクロップ順序修正について

**はい、今回のwiki 02_alignment_fix と 04_chromakey_crop にまとめられています。**

水曜夕会後に行ったのは以下の対応です。

### 何をしたか

1. **`crop_fixed_params` + 前景中心自動検出の導入**
   - テスト画像から部品の前景中心を自動検出（Otsu + closing）
   - その中心を基準にマスターと同じサイズの領域を切り出す
   - これにより撮影ごとの部品位置変動に自動で追従するようになった

2. **`estimateAffinePartial2D` への統一**
   - 画角合わせで使っていた `findHomography`(8自由度) を `estimateAffinePartial2D`(4自由度) に変更
   - シアー・透視歪みが完全に排除された

### なぜ一気に精度が上がったか

これまでのフローでは:
```
テスト画像 → 固定座標で切り出し(位置がずれる) → ホモグラフィで位置合わせ(歪む) → 比較(精度低い)
```

修正後:
```
テスト画像 → 前景中心で切り出し(位置が正確) → 相似変換で位置合わせ(歪まない) → 比較(精度高い)
```

**crop の段階で位置が正確に合っている**ため、その後の位置合わせの負荷が減り、歪みも入らなくなった。crop の精度改善が全体の精度に直結した形です。
