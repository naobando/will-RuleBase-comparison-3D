## 1. 保存機能（5d見積もり・4月中旬）

### 推奨: 年 > 月 > 日 > 部品別 > 個体別のフォルダ構成

```
検査結果/
  2026/
    04/
      14/
        平面サンプル/
          001_20260414_143052_OK/
            master.png      ← この検査で使用したマスター画像
            input.png       ← 入力画像
            result.png      ← 結果画像（BBOX描画済み）
            diff.png        ← 差分ヒートマップ
            result.json     ← 判定結果（数値・パラメータ）
          002_20260414_143105_NG/
            master.png
            input.png
            result.png
            diff.png
            result.json
          003_20260414_143120_OK/
            ...
        立体サンプル/
          001_20260414_144210_NG/
            master.png
            input.png
            result.png
            diff.png
            result.json
```

- **連番**: 日ごとにリセット。「今日何個目の検査か」が一目でわかる
- **タイムスタンプ**: 「いつ検査したか」
- **OK/NG**: フォルダ名で判定結果がわかる
- **master.png を各検査に保存**: マスター変更後でも「この検査時のマスター」が追跡できる

### JSON の中身（推奨）

```json
{
  "timestamp": "2026-04-14T14:30:52",
  "serial_number": 1,
  "part_name": "平面サンプル",
  "verdict": "OK",
  "ssim": 0.9968,
  "mse": 4.34,
  "bbox_count": 0,
  "bboxes": [],
  "processing_time_ms": 850,
  "parameters": {
    "diff_thresh": 38,
    "min_area": 4216,
    "morph_kernel": 11,
    "ensemble_thresh": 190,
    "bbox_merge_distance": 17
  }
}
```

### 設計ポイント

- **部品名はマスター登録時にユーザーが入力**。以降の検査で自動的にフォルダ分け
- **日付フォルダは自動生成**。日をまたいだら新フォルダ
- **連番+タイムスタンプ+判定のフォルダ名**。同じ部品の個体違いが区別できる
- **マスター画像を各検査に保存**。マスター差し替え前後の追跡が可能
- **パラメータも保存**。後から「この設定で検査した」が追跡できる
- **保存先はconfig.yamlで設定可能**にする（デフォルト: `検査結果/`）

## 2. 入力ゲーティング（品質チェック）

### 推奨: 撮影時に自動チェック、NG なら再撮影を促す

| チェック項目 | 判定方法 | 閾値（推奨） | 対応 |
| --- | --- | --- | --- |
| ブレ検知 | Laplacian の分散 | < 100 で警告 | 再撮影を促す |
| 露出オーバー | 白飛びピクセル比率 | > 5% で警告 | 照明調整を促す |
| 露出アンダー | 暗すぎるピクセル比率 | > 80% で警告 | 照明調整を促す |
| 部品未検出 | クロマキー前景面積比 | < 5% で警告 | 部品の配置を確認 |
| 部品はみ出し | 前景が画像端に接触 | 4辺のいずれか | 配置の調整を促す |

### 設計ポイント

- **警告は出すが検査をブロックしない**（品質ゲート = block モードはオプション）
- **警告レベルをconfig.yamlで調整可能**にする
- **結果JSONに品質チェック結果も保存**する
- 既存の `quality_gate` 機能を拡張して実装

## 3. マスター登録フロー

### 推奨: クロマキー自動切り出し + 部品名入力 + 保存

```
1. ユーザーがカメラ撮影 or ファイル選択
2. chromakey_crop で自動切り出し（既に実装済み）
3. プレビュー表示 → ユーザーが確認
4. 部品名を入力（例: 「平面サンプル」）
5. マスター画像を保存:
     master/
       平面サンプル/
         master.png              ← 切り出し済みマスター画像
         master_raw.png          ← 元の広角画像（再切り出し用に保持）
         registration_info.json  ← 切り出しパラメータ
6. config.yaml にマスター情報を記録
```

### registration_info.json

```json
{
  "registered_at": "2026-04-14T14:00:00",
  "part_name": "平面サンプル",
  "raw_image_size": [1356, 1568],
  "crop_bbox": [243, 522, 636, 948],
  "fg_center": [561, 996],
  "method": "chromakey"
}
```

### 設計ポイント

- **元画像を保持**しておくことで、パラメータ変更時に再切り出しが可能
- **部品名が保存フォルダ名になる**ため、検査結果の整理と連動
- **複数マスターの切り替え**をUIから可能にする（ドロップダウン等）

## 4. 検査結果の表示（UI）

### 推奨: 現状の2枚表示（基準+結果）+ 判定の大きな表示

| 要素 | 現状 | 推奨 |
| --- | --- | --- |
| 判定表示 | 「待機中」の文字 | OK/NG を大きく色付きで表示（緑/赤） |
| BBOX数 | 結果画像内のテキスト | 判定の下に「異常箇所: 8個」と表示 |
| SSIM | 結果画像内のテキスト | 判定の下に小さく表示 |
| 処理時間 | ステータスバー | 判定の下に小さく表示 |
| 保存ボタン | 左パネル | 結果表示後に自動保存（手動保存も可能） |

### 設計ポイント

- **自動保存をデフォルトにする**。検査員が保存を忘れない
- **履歴一覧はV1.1以降**。V1では日付フォルダを直接開く運用

## 5. 検査フロー（運用想定）

### 推奨: 最小ステップの検査フロー

```
起動 → マスター選択 → [部品を置く → 撮影ボタン → 判定表示 → 自動保存] × N個
```

### カメラ接続時の自動検査（将来）

```
起動 → マスター選択 → [部品を置く → 自動撮影 → 自動判定 → 自動保存] × N個
```

### 設計ポイント

- V1では**手動撮影（ボタン押下）**。自動撮影はV1.1以降
- **1検査あたり1秒以下**（現状達成済み）
- **連続検査に対応**。検査完了後すぐに次の部品を検査できる
- 前回の結果クリア（実装済み）

## 6. パラメータ管理

### 推奨: マスターごとにパラメータを保存

```
master/
  平面サンプル/
    master.png
    params.yaml    ← この部品専用のパラメータ
  立体サンプル/
    master.png
    params.yaml    ← この部品専用のパラメータ
```

### params.yaml の中身

```yaml
# 平面サンプル専用パラメータ（config.yaml の analysis セクションを上書き）
diff_thresh: 38
ensemble_thresh: 190
# 指定しないパラメータは config.yaml のデフォルト値を使用
```

### 設計ポイント

- **デバッグUIで調整したパラメータを部品ごとに保存**できるようにする
- config.yaml はグローバルなデフォルト値、params.yaml は部品固有の上書き
- マスター切り替え時にパラメータも自動で切り替わる

## 8. インフラ方針（推奨）

### 推奨: V1はエッジ（ローカル）で完結

| 項目 | 推奨 | 理由 |
| --- | --- | --- |
| 実行環境 | ローカルPC（Windows/Mac） | 製造業はデータ外部送信に慎重 |
| データ保存 | ローカルディスク | 検査画像の外部保存を避ける |
| アップデート | 手動（USBまたはインストーラー） | V1では十分 |
| ログ収集 | ローカルファイル | 障害時に手動で送付 |

### V1.1以降の拡張

- クラウド連携（オプション）: 検査結果の集約・ダッシュボード
- リモートアップデート: OTA配信
- リモート監視: 稼働状況の可視化

## 9. 鏡面・動的要素への対応方針

### 推奨: V1では制約として明文化、V1.1以降で対策検討

| 制約 | 内容 | 対応方針 |
| --- | --- | --- |
| 鏡面反射 | ネジ角度の変化で偽陽性 | V1では「動的要素のないワークが対象」と明記 |
| 光源映り込み | 湾曲面に光源が映る | 筐体・照明の物理的改善で対応 |
| 動的要素 | ネジ回転、可動部品 | 対象外とするか、マスクで除外する仕組みを検討 |

### 将来的な対策案（V1.1以降）

- **領域マスク**: ユーザーがマスター上で「ここは無視」領域を指定
- **回転不変特徴量**: ネジのような回転対称な要素を自動検出して除外
- **偏光フィルタ**: 物理的に鏡面反射を除去
